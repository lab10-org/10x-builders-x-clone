# Modifying Database Schema

## Objetivo
Estandarizar un flujo unico y repetible para cambios estructurales de base de datos en Supabase local, usando exclusivamente migraciones SQL versionadas.

## Flujo obligatorio (Ciclo 1-2-3)
1. **Crear el archivo (Scaffold)**
   - Ejecuta en modo no interactivo:
   - `supabase migration new <nombre_descriptivo_del_cambio> < /dev/null`
   - Resultado esperado:
   - Se crea un archivo en `supabase/migrations/<timestamp>_<nombre>.sql`.

2. **Escribir el SQL (Code)**
   - Abre el archivo generado y escribe las sentencias SQL del cambio.
   - Ejemplo:
   - ```sql
     create table public.equipos (
       id bigint generated by default as identity primary key,
       nombre text not null,
       creado_en timestamptz default now()
     );
     ```

3. **Aplicar en local (Apply)**
   - Ejecuta:
   - `supabase migration up`
   - Resultado esperado:
   - Se aplican migraciones pendientes en la instancia local.
   - Si necesitas rehacer todo desde cero (borra datos), usa:
   - `supabase db reset`

## Manejo de errores
- Si `migration up` falla por SQL invalido, corrige el mismo archivo de migracion y vuelve a ejecutar `supabase migration up`.
- Si el entorno queda inconsistente en desarrollo local, usar `supabase db reset` como ultimo recurso.

## Restricciones
- SIEMPRE usar migraciones SQL dentro de `supabase/migrations/`.
- SIEMPRE respetar el orden generar -> editar -> aplicar.
- SIEMPRE usar `supabase ...` (CLI instalado), NUNCA `npx supabase ...`.
- NUNCA hacer cambios estructurales desde la UI de Supabase.
- NUNCA usar `supabase db push` para desarrollo local.
- NUNCA mezclar en una misma migracion cambios no relacionados.

## Nota operacional (evitar comandos colgados)
- En terminales no interactivas, `supabase migration new ...` puede quedar esperando `stdin` aunque ya haya generado el archivo.
- Para evitarlo, ejecutar siempre: `supabase migration new <nombre> < /dev/null`.
- Si una ejecucion previa queda colgada:
  1. Verificar si el archivo ya existe en `supabase/migrations/`.
  2. Si existe y el proceso sigue activo, terminar ese proceso y continuar.
  3. No volver a crear la misma migracion si ya se genero el archivo.

## Cheat sheet de comandos
- Crear migracion (no interactivo): `supabase migration new <nombre> < /dev/null`
- Aplicar pendientes: `supabase migration up`
- Reset local completo (borra datos): `supabase db reset`

## Checklist de cierre
- [ ] Migracion creada con nombre descriptivo.
- [ ] SQL escrito en `supabase/migrations/<timestamp>_<nombre>.sql`.
- [ ] `supabase migration up` ejecutado sin errores.
- [ ] Si hubo error, se corrigio y se reintento `migration up`.
- [ ] No se uso UI ni `db push` para cambios estructurales.
